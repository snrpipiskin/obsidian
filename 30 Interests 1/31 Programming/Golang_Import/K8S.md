
## Что такое Kubernetes в двух словах

**Kubernetes — это оркестратор контейнеров.**

Он умеет:

- запускать твои контейнеры,
- следить, чтобы они не падали,
- раскатывать новые версии,
- масштабировать,
- управлять конфигурациями.

Фактически это как «операционная система для дата-центра».

---

## Основные сущности

### **Pod**

- Минимальная единица в K8s.
- Внутри — **один или несколько контейнеров**, которые запускаются вместе.
- Обычно = 1 контейнер = твой сервис.

✦ Собеседование: _«Что такое Pod?»_

> Pod — это набор контейнеров, которые работают вместе и делят сеть и файловую систему. Обычно в Pod’е запускают один контейнер приложения.

---

### **Deployment**

- Определяет **как запускать Pod’ы**.
    
- Сколько реплик?
    
- Как обновляться?
    
- Что делать при падении?
    

✦ Собеседование: _«В чем разница между Pod и Deployment?»_

> Pod — единичный экземпляр приложения, а Deployment управляет набором Pod’ов и обеспечивает их автозапуск, масштабирование и обновления.

---

###  **Service**

- Делает Pod’ы доступными по стабильному IP/имени.
- Потому что Pod может умереть и пересоздаться → IP изменится.
- Service абстрагирует это.

Типы:

- **ClusterIP** — внутри кластера
- **NodePort** — наружу через порт ноды
- **LoadBalancer** — наружу под облачный LB

✦ На собесе: _«Зачем нужен Service?»_

> Чтобы дать стабильную точку доступа к Pod’ам, даже если они пересоздаются и меняют IP.

---

###  **Ingress**

- HTTP/HTTPS маршрутизация внутрь кластера.
- Управляет URL-ами и доменами.
- Например: `/api` → сервис backend, `/static` → сервис frontend.

✦ На собесе:

> Ingress — это маршрутизатор для внешнего трафика, работающий поверх Services.

---

###  **ConfigMap и Secret**

- ConfigMap = не чувствительные конфиги
- Secret = чувствительные данные (пароли, токены), хранятся в base64

Обычно монтируются в контейнер через env-переменные.

---

### **Volumes**

- Хранилище файлов.
- Нужно, если контейнер должен сохранять что-то между рестартами.

Backend-разработчики часто слышат термин **PVC (Persistent Volume Claim)** — это запрос на storage.

---

### **Horizontal Pod Autoscaler (HPA)**

- Автомасштабирование Pod’ов по метрикам: CPU, RAM, custom metrics.

---
### Control Plane и Worker Nodes — архитектура Kubernetes

Kubernetes работает как распределённая система, где есть две большие группы серверов:

---

### Control Plane (управляет кластером)

Control Plane **не запускает приложения** — он:

- принимает решения,
- планирует,
- следит за состоянием,
- обеспечивает консистентность.

Состоит из четырёх ключевых компонентов:

#### **API Server**

- центральная точка входа;
- принимает все команды: `kubectl apply`, запросы контроллеров, сервисов;
- единственный компонент, с которым взаимодействуют пользователи и другие процессы.

Фактически — это REST API Kubernetes.

---

#### **etcd**

- распределённое key-value хранилище;
- содержит текущее состояние всего кластера: манифесты объектов, метаданные, события.

Если etcd умер — кластер жив, но **ничего нельзя изменить**.  
Поэтому etcd — критически важная часть.

---

#### **Scheduler**

- решает, на какой Worker Node запустить новый Pod;
- учитывает:
    - ресурсы ноды (CPU, RAM),
    - taints/tolerations,
    - node affinity/anti-affinity,
    - нагрузки,
    - ограничения.

По сути — «алгоритм, который выбирает место».

---

#### **Controller Manager**

- набор контроллеров, каждый отвечает за что-то своё:
    - DeploymentController
    - ReplicaSetController
    - NodeController
    - NamespaceController
    - И т.д.

Все контроллеры работают по принципу **reconciliation loop**:

> «Текущее состояние ≠ желаемое → привожу в соответствие»

Например:  
Deployment говорит «хочу 3 Pod’а», а в etcd есть только 2 → Controller создаёт третий.

---

### Worker Nodes (узлы, которые запускают твой код)

Worker ноды — обычные сервера (VM или физические), на которых реально работают приложения.  
На них есть три важные компонента:

#### **kubelet**

- основной агент ноды;
- получает задания от Control Plane («запусти Pod»);
- следит за здоровьем контейнеров;
- репортит в Control Plane, что происходит.

---

#### **container runtime**

Реально запускает контейнеры. Это может быть:

- containerd (стандарт сейчас),
- CRI-O,
- Docker (раньше был популярен, сейчас вытесняется).

---

#### **kube-proxy**

- компонент, обеспечивающий сетевую маршрутизацию внутри кластера;
- настраивает iptables или IPVS так, чтобы трафик попадал в нужные Pod’ы.

---

### 👇 Итог по архитектуре

> Control Plane принимает решения, Worker Nodes выполняют работу.  
> Контроллеры приводят фактическое состояние в соответствие с желаемым.

---

### Deployment vs StatefulSet

ЭТО ЧАСТО СПРАШИВАЮТ НА СОБЕСЕ.

#### Deployment

Используется для **stateless** сервисов:

- backend API
- frontend
- воркеры
- cronjobs (через CronJob)

Свойства:

- Pod’ы взаимозаменяемы
- нет фиксированных имён Pod’ов
- Storage — отрывочный (ephemeral)
- обновления через ReplicaSet и rolling update

---

#### StatefulSet

Используется для **состоящих сервисов**, которые требуют стабильности:

- базы данных: Postgres, Cassandra
- очереди: RabbitMQ, Kafka
- системы хранения

Свойства:

- фиксированные имена Pod’ов: `app-0`, `app-1`, `app-2`
- гарантия порядка запуска/остановки
- каждый Pod получает **своё** постоянное хранилище:
    - PVC для `app-0`
    - PVC для `app-1`
    - PVC для `app-2`

Это важно, если данные не должны перемешиваться.

---

### Пробы: Liveness, Readiness, Startup

Тоже обязательная тема.
#### Liveness Probe

Проверяет, жив ли контейнер.

Если liveness упала → **kubelet перезапускает контейнер**.

Используется для ситуаций типа:

- deadlock в приложении,
- зависшие воркеры,
- фатальные ошибки.

---

#### Readiness Probe

Проверяет, готов ли контейнер принимать трафик.

Если readiness «false» → Pod **убирают из Service endpoints**.

Используется:

- при старте (пока прогревается, например подключается к БД),
- при временных перегрузках,
- при maintenance внутри приложения.

---

#### Startup Probe

Нужна для **медленно стартующих** приложений (Java, Node с большим cold start).

Если Startup Probe есть — Kubernetes не будет применять liveness, пока startup не пройдет.

---

### Как устроена сеть в Kubernetes

Это тоже важная тема.

Kubernetes сеть работает по принципу:

> «Каждый Pod получает свой IP, и Pod может общаться с любым Pod’ом в кластере без NAT».

Сеть состоит из слоёв:

---

#### Pod Network (CNI)

Это плагин, который создаёт IP и сетевые интерфейсы:

Популярные CNI:

- Calico
- Flannel
- Cilium (eBPF-based, самый современный)
- Weave

CNI создаёт виртуальную сеть между Pod’ами.

---

#### Service (ClusterIP)

Service — это виртуальный IP (VIP), который маршрутизируется kube-proxy.  
Kube-proxy использует iptables или IPVS для балансировки между Pod’ами.

---

#### Ingress

Ingress — HTTP/HTTPS маршрутизация снаружи кластера внутрь.

Ingress контроллеры:

- nginx ingress
- traefik
- istio ingress gateway
- haproxy
- AWS ALB Ingress Controller

Ingress умеет:

- SSL termination,
- маршрутизацию по домену,
- маршрутизацию по пути.
## CI/CD и деплой

Тебя могут спросить:

**«Как происходит деплой приложения в Kubernetes?»**

Хороший ответ:

> Мы собираем Docker-образ, пушим его в registry. Затем CI/CD применяет манифесты (или Helm-чарты) — Deployment получает новую версию образа, Kubernetes раскатывает обновления по стратегии (например Rolling Update), создавая новые Pod’ы и удаляя старые.

---

## Как Kubernetes помогает backend-разработчику

- **автоматически перезапускает твой сервис**, если он упал;
- **гарантирует нужное число экземпляров** (реплик);
- **катит обновления без простоя**;
- **делает конфигурацию централизованной**;
- **масштабирует сервис под нагрузку**.

## Репликация

**Репликация = копирование одних и тех же данных на несколько узлов.**

Цель:

- отказоустойчивость,
- масштабирование чтений.

Примеры:

- PostgreSQL Primary → Replica,
- MongoDB replica set,
- Redis read replicas.

### Когда применяется

- много читаем (например, аналитика),
- нельзя потерять данные при падении ноды.

На собес:

> Репликация — это когда данные одинаковы на всех узлах. Улучшает отказоустойчивость и скорость чтения, но не решает проблему масштабирования записи.

---

## Шардирование

**Шардирование = деление данных на части (шарды), где каждая часть хранится на своём узле.**

Цель:

- масштабирование записи,
- масштабирование общего объёма данных.

Примеры:

- UserID 0–100k → shard1
- UserID 100–200k → shard2
- Такой подход используют: MongoDB, Elasticsearch, PostgreSQL Citus.

### Когда применяется

- слишком много данных для одной машины,
- слишком высокая нагрузка на записи,
- данные легко самому разбить.

На собес:

> Шардирование — это распределение данных по разным узлам. Каждый узел хранит свою часть данных. Это позволяет масштабировать записи и общий объём данных, но усложняет запросы и архитектуру.

---

## Репликация vs Шардирование (в одной таблице)

|Критерий|Репликация|Шардирование|
|---|---|---|
|Что происходит|копируем одни и те же данные|делим данные на части|
|Цель|отказоустойчивость, чтения|масштабирование записи и объёма|
|Недостатки|write-блоки — запись всё равно одна|сложность маршрутизации и запросов|
|Подходит для|быстрых чтений, отказоустойчивости|больших нагрузок и больших данных|

---

## Как это связывать с Kubernetes (если вдруг спросят)

Можно ответить так:

> Kubernetes масштабирует контейнеры (реплики Pod’ов), но он не решает проблему распределения данных. Для данных используются механизмы репликации и шардирования на уровне СУБД. Kubernetes просто разворачивает StatefulSet-ы с ними.


## Как реально работает Kubernetes под капотом (вся цепочка)

Ты пишешь:

`kubectl apply -f deployment.yaml`

Происходит:

1. **kubectl отправляет YAML в API Server**
2. API Server валидирует и сохраняет объект в etcd
3. Controller Manager замечает, что появился новый Deployment
4. Он создаёт соответсвующий ReplicaSet
5. ReplicaSet создаёт нужное число Pod’ов
6. Scheduler выбирает ноды для Pod’ов
7. kubelet на выбранных нодах получает команду → запускает контейнеры
8. kube-proxy обновляет маршрутизацию
9. Пробами проверяется состояние Pod’ов
10. Deployment следит за количеством реплик и состоянием rollout’а

И всё это работает бесконечно в цикле reconciliation.
