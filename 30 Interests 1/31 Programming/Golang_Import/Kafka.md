
# **Какую проблему решает Kafka?**

Современные системы состоят из десятков и сотен сервисов. Они должны обмениваться данными:

- заказы,
- платежи,
- юзерские действия,
- логирование,
- события обновления состояния.

Если сервисы общаются напрямую (через HTTP или RPC), возникают проблемы:

- тесная связанность,
- высокая нагрузка,
- отсутствие устойчивости к сбоям,
- сложно горизонтально масштабироваться,
- сложность повторной обработки событий.

**Kafka решает это, превращая систему в событийную архитектуру.**

Она предоставляет:

- надёжное хранилище событий,
- высокопроизводительный механизм публикации и потребления,
- возможность масштабного fan-out (один пишет — много читает),
- возможность хранения и воспроизведения событий.

**Kafka — это распределённый commit log для событий всей системы.**

---

# **Что такое брокер?**

**Брокер Kafka** — это отдельный сервер (или контейнер), в котором хранятся данные и который обрабатывает операции:

- принимает сообщения от producers,
- сохраняет их на диск,
- отдаёт потребителям,
- участвует в кворумах лидерства,
- управляет репликациями partition’ов.

**Kafka-кластер = несколько брокеров**, которые обеспечивают:

- отказоустойчивость,
- масштабируемость,
- высокий throughput.

# **Основные сущности Kafka**

### **Topic**

Логическая категория сообщений (например, `orders`, `payments`).  
Топик разбивается на партиции.

### **Partition**

Физическая последовательность сообщений.  
Kafka пишет в неё только append-only, что делает её безумно быстрой.

### **Offset**

Смещение сообщения в партиции.  
Используется для отслеживания прогресса потребителями.

### **Producer**

Отправляет сообщения в топик. Может выбирать:

- отправку с подтверждением,
- уровень гарантии (acks),
- ключ для партиционирования.

### **Consumer**

Читает сообщения из топика. Может управлять:

- авто- или мануальным коммитом оффсетов,
- стратегией распределения партиций.

### **Consumer Group**

Группа потребителей, разделяющих партиции топика.  
Каждая партиция принадлежит ровно одному потребителю в группе.  
Это делает Kafka очередью (queue semantics).

### **Replication**

Каждая партиция имеет:

- **leader** — принимает запросы,
- **followers** — реплицируют записи.

Используется кворум ISRs.

# **Паттерны, встроенные в Kafka**

Kafka комбинирует несколько архитектурных подходов:

### 🧩 **Distributed commit log**

Сообщения надёжно сохраняются на диск в упорядоченной последовательности.

### 📨 **Publish/Subscribe**

Один producer → множество независимых consumer-групп.

### 🔁 **Message queue semantics**

Внутри одной группы сообщения распределяются по потребителям → каждая партиция читается только одним.

### ♻ **Event sourcing**

Kafka хранит историю изменений, которую можно переигрывать (replay).

### 🧱 **Microservice decoupling**

Producer не знает, кто читатель.  
Читатель выбирает свою скорость и место чтения.

# **Гарантии доставки Kafka**

Kafka предоставляет три уровня:

### **At most once**

Сообщение может потеряться, но не будет повторено.  
Происходит, если consumer коммитит offset до обработки.

### **At least once**

Сообщение точно будет обработано, возможно повторно.  
Самая распространённая и надёжная модель.

### **Exactly once**

Kafka действительно умеет это делать (в отличие от многих MQ),  
но требует:

- идемпотентных producer’ов,
- транзакций,
- атомного коммита оффсетов и записи output-событий.

Используется редко из-за сложности, но мощная фича.

# **Особенности Kafka**

### 🔥 **Высокая пропускная способность**

Kafka оптимизирована под append-only запись и sequential reads.  
Сотни тысяч–миллионы сообщений в секунду.

### 🧊 **Хранение сообщений на диск**

Kafka устойчивее классических in-memory очередей.

### 🔁 **Retention policy**

Kafka удаляет старые данные по:

- времени (например, 7 дней),
- размеру (например, 1TB),
- compacting (хранить только последний ключ).

### 🧩 **Масштабируемость**

Увеличиваешь число партиций → увеличиваешь параллелизм.

### 🧬 **Disaster recovery**

За счёт репликации и кворумов.

### 📅 **Rewind / Replay**

Можно читать топик:

- сначала,
- с конца,
- с конкретного offset,
- с timestamp.

### 🧘 **Decoupling**

Сервисы не знают о существовании друг друга → система становится слабосвязанной.

# **Компоненты Kafka-кластера**

### **Broker** — процесс, который хранит и выдаёт сообщения.

### **Partitions** — части топика.

### **Replication** — копирование на другие брокеры.

### **ZooKeeper / KRaft** — метаданные кластера (новые версии используют KRaft).

### **Producer** — отправитель.

### **Consumer** — получатель.

### **Consumer Group** — группа, обрабатывающая топик параллельно.

### **Offsets Storage** — хранилище прогресса (обычно internal topic `__consumer_offsets`).

### **Schema Registry** — управление схемами сообщений (Avro/JSON/Proto).

# **Краткая схема процесса сообщений**

## Producer → Broker

1. Сериализация
2. Выбор партиции
3. Batch
4. Отправка лидеру
5. Запись в commit log
6. Репликация
7. ACK producer’у

## Consumer ← Broker

1. Вступление в группу
2. Получение партиций
3. Fetch сообщений
4. Обработка
5. Commit offset
