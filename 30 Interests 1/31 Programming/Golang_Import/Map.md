
Map - это сама по себе хэш таблица, представляющая из себя "ключ-значение".  
Под капотом у неё всегда есть две вещи: хэш-функция (из ключа → число фиксированной длины) и некий «массив слотов», куда это число превращается в индекс. В Go у map есть структура `hmap` (метаданные: размер, степень, флаги, указатели на буферы) и набор внутренних структур, где реально лежат ключи/значения. ### Старая реализация (до Go 1.24) — бакеты

> Под капотом в Go для мапы раньше использовали бакеты, типа с помощью хэш функции вычисляли некий хэш для ключа и размещали его в определённых бакет, брав остаток от деления на N - количество бакетов, но там были свои проблемы, из-за которых в бакете если больше 8 элементов было, то там создавался поинтер на следующий бакет и было некое вертикальное заполнение, нежели горизонтальное

Чуть точнее: количество бакетов всегда было степенью двойки: `2^B`. Вместо «остатка от деления» Go делал `hash & (2^B-1)` — это тот же `mod`, но через дешёвую побитовую маску. Каждый бакет (`bmap`) содержал **ровно 8 слотов** под ключ/значение. Если при вставке все 8 слотов были заняты, создавался **overflow-bucket** и к основному бакету цеплялся указатель — получалась вертикальная цепочка: основной бакет → overflow1 → overflow2 и т.д. [Stack Overflow+1](https://stackoverflow.com/questions/67445332/why-does-the-bucket-of-map-in-golang-contain-up-to-8-key-elem-pairs-why-the-num?utm_source=chatgpt.com)

Проблема такой схемы — когда хэши плохо распределились или мапа сильно забита, в одном бакете (и его оверфлоу) накапливается много элементов. Поиск превращается в прогулку по связному списку бакетов, с кучей прыжков по памяти → много cache-miss, ветвлений и просадка по времени. Для борьбы с этим Go следил за **load factor** (около 6.5 элементов на бакет в среднем) и при превышении начинал **расширение**: количество бакетов удваивалось, а старые постепенно «эвакуировались» в новые при последующих операциях (постепенный rehash, чтобы не было стоп-the-world на одну большую пересортировку). [dave.cheney.net+1](https://dave.cheney.net/2018/05/29/how-the-go-runtime-implements-maps-efficiently-without-generics?utm_source=chatgpt.com)

> Структура бакетов была что-то типа такой: key1key2key3value1value2value3

На уровне идеи — да: внутри бакета хранились **массив ключей** и **массив значений**, по 8 штук каждый. Реальная структура выглядела примерно так:

- массив `tophash[8]` — по 1 байту метаданных на слот;
- за ним подряд массив из 8 ключей;
- за ним подряд массив из 8 значений.

То есть в памяти это что-то вроде:  
`tophash[8] | key[0..7] | value[0..7]`.

Почему так? Чтобы **ключи** и **значения** лежали плотно и предсказуемо — это легче для GC и кэш-линии, а крошечный `tophash` позволяет быстро отфильтровать заведомо неинтересные слоты без чтения самих ключей. [i0tool5.github.io+1](https://i0tool5.github.io/learnings/golang/maps-internals/?utm_source=chatgpt.com)

---

> там был конечно некий Low Order Hash, который помогал сразу на входе в бакет понять, есть ли ключ в этом бакете, но всё равно, чтобы вытащить значение надо было заходить в бакет

В старой реализации в `tophash` писали несколько старших бит хэша (по факту 8 бит: «top hash»), плюс специальные значения для «пусто», «перемещён» и т.п. Алгоритм примерно такой:

1. Считаем полный хэш ключа.
    
2. По низким битам определяем индекс бакета (`bucketIndex = hash & bucketMask`).
    
3. Внутри бакета пробегаем массив `tophash[8]`.
    
    - Если байт `tophash[i]` не совпадает с верхними битами нашего хэша — можно сразу пропускать слот, не читая ключ.
    - Если совпал — **только тогда** лезем в массив ключей, сравниваем настоящий ключ.

Это уже хороший фильтр, но из-за overflow-цепочек всё равно приходилось следить за указателями, прыгать между бакетами и каждый раз сканировать `tophash` по 8 элементов в каждом звене цепи.

### Новая реализация (Go 1.24+) — Swiss Table style

> В свежих версиях Go от этого отказались и придумали группы из 8 элементов вместо бакетов, а так же теперь каждый элемент это 8 байт = 64 бита, и там есть некий H1 = 57 бит и H2 = 7 бит, и к H2 добавляют бит заполненности, который помогает определить, есть ли ключ в этой группе

Начиная с Go 1.24 мапы переписали на дизайн, вдохновлённый **Swiss Table** (как в absl:flat_hash_map и Rust HashMap). Теперь это уже не «бакеты с оверфлоу», а **открытая адресация с группами** (clusters) по 8 слотов и плотной мета-информацией. [go.dev+2themsaid.com+2](https://go.dev/blog/swisstable?utm_source=chatgpt.com)

Идея примерно такая:

- Полный хэш делят на две части:
    
    - **H1** (старшие ~57 бит) — используется для выбора начальной группы и дальнейшего пробирования (куда смотреть дальше при коллизиях).
        
    - **H2** (младшие 7 бит) — это сигнатура, которая хранится в **метаданных** для конкретного слота.
        
- Для каждой группы из 8 слотов есть **контрольное слово** (8 байт). Каждый байт отвечает за один слот:
    
    - 1 бит — «занят/свободен» (или спец-значения вроде tombstone);
        
    - 7 бит — тот самый H2 (top hash).
        

Что это даёт:

1. Когда мы ищем ключ, мы сначала по H1 вычисляем **индекс группы**.
    
2. Потом **одним чтением 64-битного control word** получаем состояние всех 8 слотов сразу.
    
3. Дальше можно:
    
    - либо обычным циклом по байтам,
        
    - либо SIMD-инструкциями (в C++/Rust, в Go — аналогичная идея),  
        быстро найти те слоты, где H2 совпадает.
        
4. Только после этого читаем реальные ключи/значения из памяти и сравниваем ключ полностью.
    

За счёт этого:

- меньше указателей → нет цепочек overflow-бакетов;
    
- меньше скачков по памяти → лучше cache locality;
    
- поиск чаще укладывается в пару последовательных чтений памяти (группа, потом ключи), без прыжков по произвольным адресам.
    

При коллизиях новая реализация использует **пробирование по группам** (вариация линейного / квадратичного), двигаясь по таблице блоками по 8 слотов, пока не найдёт пустое место или нужный ключ. Это сильно уменьшает «вертикальное» накопление как раньше в overflow-цепочках — заполнение теперь идёт **горизонтально по таблице**, более равномерно.

### Потокобезопасность

> Сама по себе map не потокобезопасная, нельзя использовать мапу из нескольких горутин без синхронизации, нужна SyncMap

Это не поменялось ни в старой, ни в новой реализации: **обычная `map` в Go не потокобезопасна**. Если из нескольких горутин одновременно и читаем, и пишем в одну и ту же map — нужно:

- либо оборачивать её в `sync.Mutex` / `sync.RWMutex` и всегда работать под локом;
    
- либо использовать `sync.Map`, но она оптимизирована под специфические кейсы (много читателей, редкие записи, долгоживущие ключи). Для «обычных» структур данных часто лучше `map + RWMutex`, чем слепо ставить `sync.Map`. [Habr](https://habr.com/ru/companies/avito/articles/774618/?utm_source=chatgpt.com)
