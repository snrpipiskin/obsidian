

# Что такое context

объект, предназначенный для внешнего контроля над операциями, а также для передачи информации

# 🔍 Зачем нужен `context`

Главная идея: **управлять временем жизни операций и передавать “сигналы” вниз по стеку вызовов.**


Контекст решает три задачи:

1. **Отмена** (`cancelation`) — возможность остановить все дочерние горутины, если, например, пользователь закрыл соединение, произошла ошибка или вышел таймаут. (context.withCancel)
2. **Таймауты и дедлайны** — можно задать время, после которого операция должна прекратиться. (`ctx, cancle := context.withTimeout(contex.Background(),  15 * time.Second)` или context.withDeadline)
3. **Передача значений** (ключ–значение) — например, `requestID`, `userID`, `traceID` и т.д. (но _не для данных_, а для метаданных запроса). (context.withValue) - но это антипаттерн

---

# Основные функции и типы

context.Background()  // базовый корень для всего приложения (обычно корневной контекст)
context.TODO()  // “пока не знаю, какой контекст передать” 
context.WithCancel(parent)
context.WithTimeout(parent, duration)
context.WithDeadline(parent, time.Time) context.WithValue(parent, key, value) //а так же используется под капотом у withTimeout

Каждая из функций создаёт **новый контекст**, наследующий родительский.

---

# Как это работает под капотом

Контекст — это просто интерфейс:

`type Context interface {     
Deadline() (deadline time.Time, ok bool)     
Done() <-chan struct{}     
Err() error    
Value(key any) any
}`

- `Done()` возвращает **канал**, который закрывается, когда контекст отменяется.
- `Err()` возвращает причину отмены (`context.Canceled` или `context.DeadlineExceeded`).
- Отмена или дедлайн распространяются **вниз по дереву** всех потомков (цепочка родитель→дочерний).
- Всё реализовано без тяжёлых блокировок, на каналах и атомиках.


# Как работает отмена контекста по deadline/timeout

Когда ты делаешь:

`ctx, cancel := context.WithTimeout(parent, 3*time.Second)`

Go создаёт **timerCtx**:

- в нём хранится время, когда нужно отменить контекст — deadline,
- создаётся реальный таймер (`time.Timer`),
- таймер запускается в отдельной лёгкой горутине,
- когда он срабатывает → вызывает `cancel()`.

### 🔹 Что делает `cancel()`?

`cancel()`:

1. ставит внутренний флаг **done** в состояние “закрыто”;
2. закрывает канал `ctx.Done()` (канал закрывается ОДИН РАЗ);
3. вызывает cancel для _детей_ — рекурсивно;
4. снимает ссылку на функции отмены (prevent leaks);
5. останавливает таймер (если он есть).

**Закрытие канала `Done()` — главный механизм уведомления о прекращении работы.**

---

# 🟦 Как контекст сообщает отмену под капотом

Каждый context имеет:

`Done() <-chan struct{}`

Когда `cancel()` вызывается:

- канал `done` **закрывается**,
- это мгновенно пробуждает:    
    - `select { case <-ctx.Done(): ... }`,
    - `WaitGroup`-подобные конструкции.


Канал закрывается **без отправки**, просто закрытие.

Это важно:

> **Закрытый канал мгновенно читается всеми слушателями.**

Это и есть механизм отмены.

---

# 🟦 Теперь про каскадирование контекста (как это работает)

Когда ты создаёшь дочерний context, он хранит ссылку на родителя.

Пример:


parent := context.Background() 
ctx1, cancel1 := context.WithCancel(parent) 
ctx2, cancel2 := context.WithTimeout(ctx1, time.Second) 
ctx3 := context.WithValue(ctx2, "key", 123)

Структура теперь такая:

`Background   
└── ctx1 (cancelCtx)         
		└── ctx2 (timerCtx)               
				└── ctx3 (valueCtx)`

## 🔹 Если отменяется ctx1

- ctx1.Done() закрывается
- Go рекурсивно вызывает cancel у всех потомков:
    - ctx2 → ctx3

## 🔹 Если истечёт timeout у ctx2

- закрывается ctx2.Done()
- отменяется **только ветка ниже него**:
    - ctx3

ctx1 и родительский Background не затрагиваются.

То есть:

> **Отмена идёт вниз → к детям, но не вверх → к родителям.**

# Что происходит в коде обработчиков

Обычно код выглядит так:

`select { 
	case <-ctx.Done():     
		return ctx.Err() 
	case msg := <-ch:     
		process(msg) 
}`

Когда контекст отменён:

- `<-ctx.Done()` сразу срабатывает,
- обработчик выходит,
- ресурсы освобождаются.

# Практические сценарии

|Сценарий|Пример|
|---|---|
|⛔️ Прерывание долгих операций|API-обработчик HTTP, который отменяет работу горутин, если клиент разорвал соединение|
|🕐 Таймауты|Вызов RPC/HTTP-клиента с лимитом времени|
|🧵 Координация горутин|Запуск группы горутин, которые все должны остановиться при первом сбое|
|🧩 Трейсинг / логирование|Передача requestID или traceID вглубь сервиса|

#  Пример: HTTP-сервер

`func handler(w http.ResponseWriter, r *http.Request) {`
	`ctx := r.Context() // встроенный контекст запроса`      
	`data, err := fetchData(ctx)`     
	`if err != nil {`         
		`http.Error(w, err.Error(), http.StatusRequestTimeout)`         
		`return`     
	`}`     
	`fmt.Fprintln(w, data)`
`}`  

`func fetchData(ctx context.Context) (string, error) {`     
	`req, _ := http.NewRequestWithContext(ctx, "GET", "https://api.example.com", nil)`    
	`resp, err := http.DefaultClient.Do(req)`     
	`if err != nil {`         
		`return "", err`     
	`}`    
	 `defer resp.Body.Close()`     
	 `b, _ := io.ReadAll(resp.Body)`     
	 `return string(b), nil` 
`}`

Если клиент закрывает соединение — `r.Context().Done()` закрывается, и `http.Client` отменяет запрос.

# Как контекст помогает с горутинами

Например, “работник, который должен завершиться при отмене”:

`func worker(ctx context.Context, id int) {     
	for {         
		select {         
			case <-ctx.Done():             
				fmt.Println("worker", id, "stopped:", ctx.Err())             
				return
			default:             
				doWork()         
		}     
	} 
}`

Так мы избегаем **утечек горутин** — все подчинённые останавливаются, если родительский контекст отменён.

---

# Частые ошибки

1. ❌ Использовать `context.WithValue` для передачи бизнес-данных — **только метаданные!**
2. ❌ Не вызывать `cancel()` → утечка таймеров и горутин.
3. ❌ Игнорировать `ctx.Err()` и `ctx.Done()` внутри долгих операций.
4. ❌ Создавать контекст без родителя (`Background`) слишком глубоко — теряется связь с запросом.
