[Redis](http://redis.io/) (Remote Dictionary Service) — это опенсорсный сервер баз данных типа ключ-значение.

## Какие типы данных поддерживает

- Строка (String)
- Битовый массив (Bitmap)
- Битовое поле (Bitfield)
- Хеш-таблица (Hash)
- Список (List)
- Множество (Set)
- Упорядоченное множество (Sorted set)
- Геопространственные данные (Geospatial)
- Структура HyperLogLog (HyperLogLog)
- Поток (Stream)

## Для чего нужен

Самые частые сценарии:

1. **Кэш перед основной БД**
    - Храним часто читаемые данные: профили, настройки, ответы API.
    - Схема: клиент → Redis (кэш) → PostgreSQL (источник истины).
2. **Сессии и авторизация**
    - Сессии веб-приложений, токены, чёрные списки токенов и т.п.
    - Причина: читать/писать сессии нужно быстро и часто.
3. **Rate limiting / throttling**
    - Ограничения типа: "не более 100 запросов в минуту на IP".
    - Увеличиваем счётчик в Redis, TTL — и всё.
4. **Очереди задач / фоновые джобы**
    - Лист/стрим как очередь задач для воркеров.
    - Аналог примитивной версии RabbitMQ/Kafka.
5. **Pub/Sub**
    - Каналы для сообщений между сервисами: чат, уведомления и т.п.
6. **Краткоживущие данные**
    - Коды подтверждения (OTP), ссылки на reset пароля, временные ключи.
7. **Лидерборды, рейтинги**
    - `sorted set` — идеально: `ZADD leaderboard score userId`.
8. **Распределённые блокировки**
    - Примитивы для lock’ов между несколькими сервисами (например, Redlock).

## Почему Redis быстрее запросов в БД?

Важно: **он быстрее не потому, что “лучше”, а потому что решает другую задачу** и делает это проще.

### Всё в памяти

- Redis хранит данные в **RAM**, а не на диске (диск используется только для периодического сброса/логов).
- Доступ к памяти на порядки быстрее дискового I/O.
- Postgres каждый запрос:
    - тянет данные со страницы (RAM, но через свои буферы и кэш-слой),
    - может трогать диск,
    - делает проверки, планирование запроса, индексы и т.п.

### Простая модель данных

- В Redis запрос обычно очень простой: `GET`, `SET`, `HGET`, `LPUSH` и т.д.
    
- Нет:
    
    - сложного парсинга SQL,
    - оптимизации плана запроса,
    - JOIN’ов и агрегатов (по крайней мере в классическом стиле).

Если сравнивать:

`SELECT * FROM users WHERE id = 123;`

и

`GET user:123`

то второй объективно проще для машины.

## Но почему тогда вообще нужен PostgreSQL?

Потому что Redis **не заменяет** нормальную БД в большинстве кейсов:

**Redis**:

- не гарантирует такую же надёжную **долговечность данных** (durability),
- не про сложные **отношения** между сущностями (JOIN’ы, foreign keys),
- нет нормального SQL, сложных запросов, аналитики,
- данные можно потерять при крэше, если не настроена и не гарантирована запись на диск с нужными настройками.

**PostgreSQL**:

- ACID-транзакции, строгая консистентность.
- Схема, типы, ограничения (`NOT NULL`, `UNIQUE`, `FOREIGN KEY`).
- SQL: фильтры, сортировки, группировки, подзапросы, джоины.
- Индексы разных типов (btree, gin, gist…).
- Надёжное хранение на диске.

Поэтому обычно делают так:

- **Postgres — источник истины** (source of truth).
- **Redis — быстрая прослойка** (кэш/очереди/блокировки/сессии).


## Redis как брокер сообщений

### Какие механизмы Redis предоставляет для сообщений

Redis поддерживает **три основных способа обмена сообщениями**:

### **Pub/Sub (классический publish-subscribe)**

Простейший механизм:

- издатель посылает сообщение в канал,
- все подписчики получают его мгновенно.

Пример:

`PUBLISH news "Hello!" SUBSCRIBE news`

#### Ограничения Pub/Sub

- **Нет долговечности сообщений (durability)** — если подписчик отключился, он пропустил сообщение.
- **Нет буфера** — Redis не сохраняет историю.
- **Нет подтверждений** — нельзя понять, что сообщение доставлено.

**Вывод:** годится только для случаев, где _потеря пары сообщений не критична_. Например: обновление UI, уведомления внутри кластера, неважные real-time события.

---

### **Lists + BLPOP/BRPOP (очереди задач)**

Почти примитивная очередь:

- `LPUSH queue item`
- `BRPOP queue` — блокирующее чтение (worker ждёт появления элемента)

#### Преимущества

- Простой и быстрый механизм очередей.
- Работает как **work queue** (каждое сообщение получает ровно один воркер).

#### Недостатки

- Нет повторов/ретраев по умолчанию.
- Если воркер умер во время обработки, сообщение **потеряно**.
- Нет ack/nack.

**Вывод:** можно использовать как простую очередь, но нужно допиливать своё поведение.

---

### **Redis Streams (современный механизм для очередей Kafka-like)**

С 2018 года Redis получил новый мощный тип: **streams**.

Это уже полноценный механизм сообщений:
- **персистентность** (хранятся на диск)
- **группы потребителей (consumer groups)**
- **ack/nack**
- каждый потребитель получает “свою” часть сообщений — как в Kafka и RabbitMQ
- гарантии доставки: **at-least-once**

Команды:

`XADD mystream * message "Hello!" XREAD GROUP workers myconsumer STREAMS mystream > XACK mystream workers message-id`

### Streams позволяют:

- распределять сообщения между воркерами,
- не терять их — даже если воркер упал,
- делать реплей сообщений,
- работать с backpressure (стрим растёт пока воркеры не успевают).

### Ограничения Streams:

- не такие мощные как Kafka (нет распределённого лога из коробки),
- хранятся в памяти + на диске → при большом объёме дорожает,
- нет сложных гарантий exactly-once.

**Вывод:** **Streams — лучшее средство для брокера в Redis**, если хочется баланс между лёгкостью и надёжностью.

---

### Когда Redis — хороший брокер сообщений

Redis подходит, когда важны:

- **простота**
- **скорость**
- **минимальные накладные расходы**
- сообщения **не гигантские** (до нескольких KB)
- нужен **легковесный брокер** без сложных гарантий

Хорошо подходит для:

- очередей фоновых задач
- лёгких микросервисов
- real-time событий
- чатов/уведомлений
- event sourcing в мини-системах
- rate limiting, метрик и heartbeat-логики

---

### Когда Redis нельзя использовать как брокер сообщений

Если нужен полноценный messaging system с высокими гарантиями:

|Требование|Redis pub/sub|Streams|RabbitMQ|Kafka|
|---|---|---|---|---|
|Durability|❌|✔️|✔️|✔️|
|Ack/nack|❌|✔️|✔️|✔️|
|Replay|❌|✔️|✔️ (limited)|✔️|
|Exactly-once|❌|❌|⚠️ сложно|✔️|
|Очень большой объём сообщений|❌|⚠️|⚠️|✔️|
|Хранение терабайт логов|❌|❌|❌|✔️|

Если у тебя:

- миллионы сообщений в минуту,
- требование строгой доставки,
- аналитика событий,
- большие очереди,

то Redis как брокер — **не лучший выбор**.

---

### Вывод

### ✔ Да, Redis можно использовать как брокер сообщений.

### Но:

- **Pub/Sub** → подходит только для “легких” real-time событий.
- **Lists** → простая очередь, но без гарантий.
- **Redis Streams** → мощный, устойчивый брокер с реализацией consumer groups и ack/nack.

➡ **Для большинства небольших/средних систем Redis Streams — более чем достаточен.**

➡ **Для крупной инфраструктуры с большими очередями — лучше RabbitMQ/NATS/Kafka.**
