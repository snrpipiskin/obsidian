#### Atomics 

Атомики в Golang — это примитивы синхронизации из пакета `sync/atomic`, которые позволяют выполнять операции с памятью атомарно, то есть неделимо и безопасно в многопоточной среде. Они используются для реализации низкоуровневых операций над общими переменными без использования мьютексов, обеспечивая порядок доступа и предотвращая состояния гонки ([race conditions](https://www.google.com/search?q=race+conditions&sca_esv=9beffc660237a68f&sxsrf=AE3TifPsC0MAMbtfAgZsaVz23AkD21O1cA%3A1762610112845&ei=wEsPabOiM9eni-gP5sWJyAM&ved=2ahUKEwi0k7Dc2-KQAxWChv0HHUBXEScQgK4QegQIARAB&uact=5&oq=%D0%B0%D1%82%D0%BE%D0%BC%D0%B8%D0%BA%D0%B8+golang+%D1%8D%D1%82%D0%BE&gs_lp=Egxnd3Mtd2l6LXNlcnAiHNCw0YLQvtC80LjQutC4IGdvbGFuZyDRjdGC0L4yBhAAGBYYHjIIEAAYgAQYogQyCBAAGIAEGKIEMggQABiABBiiBEjWGFAAWKoXcAF4AJABAZgBsAOgAYwUqgEJMC45LjIuMS4xuAEDyAEA-AEBmAINoALfEcICChAjGIAEGCcYigXCAgUQABiABMICCxAuGIAEGNEDGMcBwgIFEC4YgATCAgsQLhiABBjHARivAcICFBAuGIAEGJcFGNwEGN4EGOAE2AEBwgIEEAAYHsICBhAAGAUYHsICBRAAGO8FmAMAugYGCAEQARgUkgcJMS45LjIuMC4xoAfeb7IHCTAuOS4yLjAuMbgH2hHCBwYwLjMuMTDIBzM&sclient=gws-wiz-serp&mstk=AUtExfBw3iKsXMurziKuUXmPsW-QxzXbTu_Jyhup4b6JO2OE_GFDZODfmhiSMJSd9kBAqyMLKacT85JBjQxVvNAIUc37GjBFqzm_6EEqniIgD7purcDmS1G18zyjphltQqylrU4l0iwZiXAJBNME1bvrc-1mYb3wxzfZYqLTmP30N4IAWPZU5xKr5aETYKAyLZdWn5bwNxsoat7tTLAqKdMxiVI7I8XBtoWt_31VsiUwJBsg5KcA2Hghzg7r5r_e8fZzGdc60wt_Eft4yN46EmN8KewC&csui=3)).

Методы atomics
- `Load` — атомарно прочитать значение.
- `Store` — атомарно записать новое значение.
`atomic.StoreInt64(&x, 10) val := atomic.LoadInt64(&x)`

Без `atomic`, обычное чтение/запись может привести к **гонке данных**, если параллельный поток меняет ту же переменную.

`Add` — атомарное прибавление (или вычитание, если отрицательное число):
`atomic.AddInt64(&x, 1)   // x++ atomic.AddInt64(&x, -1)  // x--`
Возвращает **новое значение** после операции.


`Swap` — атомарно **заменяет** значение на новое и **возвращает старое**:

`old := atomic.SwapInt64(&x, 99) fmt.Println("было", old, "стало", atomic.LoadInt64(&x))`

Полезно, если нужно «обнулить» счётчик и узнать, сколько там было.

`CompareAndSwap` (CAS) — сравнивает значение и меняет его **только если текущее совпадает с ожидаемым**.

`ok := atomic.CompareAndSwapInt64(&x, 10, 20) if ok {     fmt.Println("успешно заменили 10 → 20") } else {     fmt.Println("не совпало, x =", atomic.LoadInt64(&x)) }`

⚙️ Применяется в **lock-free** структурах, очередях, или когда нужно сделать что-то вроде:

> «Если флаг ещё не установлен — установи его, иначе ничего не делай».


- **Атомики** “помогают избегать гонок” только для **одного конкретного атомарного слова** (или набора слов, если вы сами аккуратно спроектировали протокол вокруг CAS). Они **не делают** операции над map/slice безопасными сами по себе.
- **Мьютекс** защищает **любой** набор операций — хоть над map, хоть над slice, хоть над несколькими связанными структурами — потому что он **серийно** пропускает горутины через критическую секцию.


#### Семафор (Semaphore)

В рамках данного подхода создаётся, по сути, неограниченное количество горутин. Их выполнение блокируется с помощью объекта Semaphore, который ограничивает количество одновременно выполняемых горутин с помощью буферизированного канала. В данном подходе на каждую задачу создаётся горутина. Однако если буфер канала семафора переполнен, то при операции Acquire горутина с задачей блокируется до тех пор, пока буфер не освободится с помощью операции Release.

Для сбора результатов используется канал, из которого происходит чтение в основной горутине (здесь, конечно, речь идёт о той, в которой выполняется функция, это совсем не обязательно может быть горутина, в которой выполняется функция main).


#### Worker pool

**Проблема:**
ты запускаешь сотни или тысячи задач (например, обработка файлов, запросов, пользователей).  
Если сделать на каждую задачу отдельную горутину — система может захлебнуться по памяти и планировке.

**Решение:**  
создать **фиксированное количество горутин** (воркеров), которые будут **брать задачи из очереди** и выполнять их.

Так мы:
- ограничиваем параллелизм;
- используем горутины повторно (меньше накладных расходов);
- контролируем нагрузку на CPU/сеть.

##### Общая схема

               +----------------+
               |   main (планировщик)  |
               +----------------+
                       |
                       v
           +----------------------+
           |   channel of jobs    | ← сюда кладём задачи
           +----------------------+
            |     |     |     |
            v     v     v     v
         worker worker worker worker


Каждый **worker**:

- слушает канал задач (`jobs <-chan Job`)
- выполняет задачу
- сигналит о завершении (например, через `WaitGroup`)

## 🧠 Пример кода (самый базовый)

package main

import (
    "fmt"
    "sync"
    "time"
)

// Задача (job)
func worker(id int, jobs <-chan int, wg *sync.WaitGroup) {
    defer wg.Done()
    for j := range jobs {
        fmt.Printf("worker %d started job %d\n", id, j)
        time.Sleep(time.Second) // имитация работы
        fmt.Printf("worker %d finished job %d\n", id, j)
    }
}

func main() {
    jobs := make(chan int, 100) // очередь задач
    var wg sync.WaitGroup

    numWorkers := 3
    for i := 1; i <= numWorkers; i++ {
        wg.Add(1)
        go worker(i, jobs, &wg)
    }

    // Отправляем задачи
    for j := 1; j <= 9; j++ {
        jobs <- j
    }
    close(jobs) // сигнал: задач больше не будет

    wg.Wait() // ждём всех воркеров
}


**Что произойдёт:**
- создаются 3 воркера;
- задачи (1–9) кладутся в канал;
- воркеры берут по одной задаче, выполняют, и берут следующую;
- задачи распределяются автоматически, примерно поровну.

##### Плюсы такого подхода
- Контролируем **максимальное количество параллельных горутин**.
- Можно легко **менять размер пула** (число воркеров).
- Работает для любых задач: HTTP-запросы, I/O, вычисления и т.д.
- Легко добавить **отмену через контекст** или **таймауты**.

##### Итого

|Что|Для чего|Аналог|
|---|---|---|
|Worker pool|Ограничить число активных горутин|thread pool|
|Каналы|Очередь задач / результатов|очередь сообщений|
|WaitGroup|Дождаться завершения всех воркеров|join()|
|Context|Управление жизненным циклом|отмена, таймауты|
