Docker — это система, которая позволяет запускать приложения в изолированных средах, называемых контейнерами.  
Контейнер — это как маленькая, самодостаточная среда, где есть всё, что нужно для работы программы: нужная версия языка, библиотеки, зависимости.

## Какие проблемы Docker решает

### 1. Проблема «у меня работает, а у вас нет»

На обычных серверах разные версии библиотек, языков, системных пакетов. Приложение может вести себя по-разному. 
Docker упаковывает всё нужное внутрь контейнера — и поведение становится стабильным.

### 2. Лёгкая переносимость между серверами

Контейнер можно запустить:

- на локальном компьютере,
- на сервере,
- в облаке.

И всё будет идентично.

### 3. Быстрое масштабирование

Нужно больше копий приложения — Docker позволяет очень быстро стартовать новые контейнеры, потому что они создаются из образа (template), который уже готов.

### 4. Изоляция

Если на сервере работает сразу много сервисов, Docker полностью изолирует их друг от друга:

- отдельные процессы,
- отдельная файловая система,
- свои переменные окружения,
- свои сети.

Приложения не мешают друг другу.

### 5. Простота сборки среды разработки

Вместо «поставь это, установи то», разработчик просто запускает контейнер — и получает рабочее окружение за секунды.

## Основные термины

**Контейнер** — это **процесс**, который запускается в изолированной среде, со своей файловой системой и виртуальной сетью, но использует ядро хостовой ОС. Он выглядит как отдельная мини-система, но не имеет полноценной ОС внутри.

---

**Image** — это **шаблон файловой системы**, который описывает, что должно быть внутри контейнера: библиотеки, зависимости, директории, конфигурация и т.д.  
Образ состоит из слоёв, которые можно переиспользовать.  
Контейнер запускается именно _из_ образа, как процесс на его основе. Образ — неизменяемый (immutable).

---

**Dockerfile** — это **инструкция**, по которой собирается image. 

В нём пошагово описано:
- какую базу брать (например, `FROM node:18`),
- какие файлы копировать,
- какие пакеты устанавливать,
- какую команду запускать внутри контейнера.

**Dockerfile** — это «рецепт», **image** — готовое «блюдо».

---

**Docker Engine / Docker Daemon** -  это фоновый сервис (демон), который работает на сервере и:

- запускает/останавливает контейнеры,
- управляет образами и сетями,
- общается через Docker API.

Все контейнеры создаются и контролируются именно через Docker Engine.

---

**Registry** - это **хранилище образов**. Например Docker Hub или GitHub Container Registry

Сюда push’ят образы, отсюда pull’ят, когда разворачивают.

---

**Volume** - это **постоянное хранилище данных**, которое живёт вне контейнера.  
Нужно для баз данных, логов, файлов, которые должны переживать перезапуск контейнера

**Контейнер** — временный, **volume** — долговечный.

---

**Network** - это виртуальная сеть Docker, в которой работают контейнеры.  
Даёт:

- собственные IP,
- изоляцию,
- возможность контейнерам общаться между собой,
- маппинг портов наружу.

---

Layer (Слой)

Образ состоит из слоёв — каждый шаг Dockerfile создаёт новый слой. 

**Слой — это неизменяемый кусок файловой системы**, созданный одним шагом Dockerfile.  
Каждый слой добавляет или изменяет файлы поверх предыдущих, а все слои вместе формируют образ.

Слои можно переиспользовать: если несколько образов имеют одинаковые слои, Docker хранит их только один раз и накладывает друг на друга как «бутерброд».

---

**Tag** - это «ярлык» для образа: например, `myapp:1.0.0` или `node:18-alpine`.  
Позволяет иметь несколько версий одного образа.


## Основные команды

#### **Запуск контейнера**

`docker run image_name` (example: docker run ubuntu)

#### **Запуск контейнера в интерактивном режиме**

*Контейнер не упадёт, даже если нет приложения внутри*

`docker run -it image_name`

#### **Запуск контейнера в detach режиме**

*Контейнер запустится в фоне, вернётся его айди (container_id)*

`docker run -d image_name`

*Тогда, чтобы в него попасть надо использовать*

`docker attach container_id`

#### **Остановка контейнера**

`docker stop container_id`

а вновь запустить командой

`docker start container_id`

#### **Посмотреть список запущенных контейнеров**

`docker ps
`
#### **Посмотреть список всех контейнеров**

`docker ps -a`

#### **Посмотреть все образы**

`docker images`

#### Переадресация портов во время запуска

`docker run -d -p 8888:80 nginx`

80 - порт nginx внутри контейнера
8888 - порт машины, на котором запущен докер
делается это для того, чтобы иметь доступ к портам внутри контейнера

## Настройка докер файла (Dockerfile)

### **FROM**

`FROM node:18`

Базовый слой образа.  
Любой Dockerfile начинается с **FROM** — задаёт “из какой среды” будет собираться образ.

---

### **WORKDIR**

`WORKDIR /app`

Устанавливает рабочую директорию внутри контейнера.  
Все последующие команды (COPY, RUN и т.д.) выполняются относительно неё.

---

### **COPY**

`COPY . /app`

Копирует файлы **с хостовой машины внутрь образа**.  
Очень часто используется для копирования исходников проекта.

---

### **ADD**

`ADD . /app`

Похож на COPY, но:

- умеет скачивать файлы по URL
- умеет распаковывать архивы

Используется **реже**, потому что ведёт себя «магически».  
Обычно рекомендуют COPY, если не нужны дополнительные фичи.

---

### **RUN**

`RUN npm install`

Запускает команду _во время сборки образа_ и фиксирует результат в новый слой.  
Применяется для установки зависимостей, пакетов, настройек окружения.

---

### **CMD**

`CMD ["node", "server.js"]`

Команда, которая запускается **при старте контейнера**.  
Если контейнер создан, но не запускается — это то, что стартует по `docker run`.

Важно:

- CMD можно переопределить при запуске контейнера.
- Бывает в двух форматах:
    - shell: `CMD node server.js`
    - exec: `CMD ["node", "server.js"]`


Exec — предпочтителен (точная передача аргументов, без shell).

---

### **ENTRYPOINT**

`ENTRYPOINT ["python3"]`

Тоже команда для запуска контейнера, но:  
**ENTRYPOINT нельзя переопределить обычным образом.**

Часто комбинируется с CMD:

`ENTRYPOINT ["python3"] CMD ["app.py"]`

Тогда реальная команда будет:  
`python3 app.py`

---

### **EXPOSE**

`EXPOSE 3000`

Документирует, какие порты контейнер _планирует_ слушать.  
Сами порты не открывает.  
Полезно для понимания инфраструктуре.

---

### **ENV**

`ENV PORT=3000`

Устанавливает переменные окружения внутри контейнера.  
Доступны в приложении как обычные environment variables.

---

### **ARG**

`ARG NODE_ENV=production`

Переменные **только на этапе сборки** (не попадают внутрь контейнера).  
Передаются через:  
`docker build --build-arg NODE_ENV=dev`

Используются для параметризации Dockerfile на build-этапе.

---

### **VOLUME**

`VOLUME /data`

Объявляет, что эта директория должна жить вне контейнера (в volume).  
Docker автоматически создаст volume, если его нет.

---

### **USER**

`USER node`

Задаёт пользователя, от имени которого будут выполняться следующие команды.  
Важно для безопасности — не запускать приложение от root.

---

### **HEALTHCHECK**

`HEALTHCHECK CMD curl -f http://localhost/health || exit 1`

Описывает, как понять, что контейнер “живой”.  
Используется оркестраторами (Docker Swarm, Kubernetes).

---

### **ONBUILD**

`ONBUILD RUN npm install`

Инструкция, которая **не выполняется в этом образе**, но будет выполнена, когда на его основе сделают другой образ.  
Используется редко (в основном — в базовых образах).

---

### **LABEL**

`LABEL maintainer="rostislav" LABEL version="1.0"`

Метаданные об образе.  
Удобно для автоматизации/CI/CD.

## Виртуализация vs Контейнеризация
### **Виртуализация**

Когда ты запускаешь виртуалку (VM), ты создаёшь **полную копию компьютера**:

- своё виртуальное железо (CPU, RAM, диск)
- своё ядро ОС
- своя полностью изолированная операционная система
- запускаемые приложения живут внутри этой ОС

Это требует много ресурсов — VM тяжёлая, медленно запускается, дублирует ядро.

---

### **Контейнеризация**

Контейнер _не имеет собственной ОС_.  
Он использует **ядро хостовой ОС**, но делает изоляцию процессов.

Контейнер — это:

- обычный процесс Linux,
- у которого своя файловая система (image),
- свои namespace (изоляция),
- свои ограничения ресурсов (cgroups).

Запуск контейнера = запуск процесса → быстро, легко и мало потребляет ресурсов.

---

### Архитектурная разница — ключевая картинка (словами)

#### Виртуальная машина (VM)

`Хард → Гипервизор → Гостевая ОС → Приложения`

#### Контейнер

`Хард → Хостовая ОС → Docker → Контейнеры (процессы)`

Разница в том, что гостевая ОС весит гигабайты, а контейнер — это только слои файловой системы и один процесс.

---

### Что именно «тяжёлое» в виртуалке

### Виртуальная машина имеет:

- **полноценную операционную систему** (ядро + systemd + init + драйверы)
- виртуальные устройства: диск, видеокарта, сеть, процессор
- загрузку ядра при старте (boot)
- полноценный системный образ 10–30+ ГБ
- замедление за счёт гипервизора

То есть VM — это как второй компьютер внутри твоего компьютера.

---

### Что именно «лёгкое» в контейнере

Контейнер имеет:

- **отдельную файловую систему (image)**
- отдельное пространство процессов (_namespace pid_)
- отдельную сеть (_namespace net_)
- изоляцию файловой системы (_namespace mount_)
- лимиты на ресурсы (_cgroups_)

Но:

- _не имеет своего ядра_
- _не имеет виртуального железа_
- _не загружается как ОС_

Контейнер — это просто процесс, но «обманутый», что он один на целой машине.

---

### Почему контейнеры стартуют за миллисекунды

Потому что контейнер — это обычный процесс Linux.

- нет boot-процесса ОС
- нет инициализации systemd
- нет виртуального железа
- нет гипервизора

Это одна из причин популярности Docker.

---

### Почему виртуалки нужны, несмотря на Docker

Есть задачи, где VM выигрывают:

### Когда нужна:

- **другая операционная система**, например запуск Linux на Windows
- **полная изоляция безопасности**
- **чёткое выделение ресурсов**
- запуск приложений, которые требуют **полного окружения с ядром**
- работа с разными ядрами (например разные версии Linux kernel)

Docker не решает эти задачи, потому что делит ядро хоста.

---

### Почему Docker удобнее для разработки и продакшена

Преимущества контейнеров:

- лёгкие
- мало памяти
- быстро стартуют
- легко масштабируются
- инфраструктура как код (Dockerfile)
- гарантированное одинаковое окружение
- идеальны для микросервисов

Для разработки:

«Запустил и забыл» — у всех одинаковая среда.

Для продакшена:

Можешь поднять 100 контейнеров из одного образа за секунды.

---

### В чём обман, когда говорят «контейнер — это лёгкая виртуалка»

Контейнер — **не виртуалка вообще**.  
Это лишь **процесс**, запущенный:

- в другой файловой системе
- в изолированном пространстве
- с заданными ресурсами

Он _ведёт себя_ как виртуалка, но архитектурно — это процессы в той же ОС.
