
Канал - структура (тип) данных, у которого основная цель -- синхронизация между горутинами. Канал сам по себе делится на буфер. и не буфер. канал
Буферизированный канал под капотом имеет буфер (круговую очередь), а так же sendx и receivex, индексы, которые указывают, в каком месте очереди мы читаем или записываем. Например изначально канал пустой и буфер длиной 4. Sendx и Receivex смотрит на нулевой индекс. После того, как произошла запись в канал, то sendx двигается на 1, и так до тех пор, пока мы не дойдём до конца. Как только дошли до конца и записали, идём в начало и при следующей записи если оттуда ещё не прочитали, то блокируем горутины и там под капотом функции gopack() используется, которая заставляет "заснуть" горутину, которая попыталась записывать в переполненный канал, а потом уже когда значение освободиться, то через goready() проснётся горутина и будет ок
Если же канал не буфер. то там вроде сразу напрямую общаются горутины и передают данные (когда пробуждаются)

Аксиомы канала:
1. читать из закрытого канала можно, но будет nil (вроде)
2. писать в закрытый канал приведет к панике
3. писать в пустой канал приведёт к блокировке горутины
4. читать из пустого канала приведёт к блокировке горутины

# **ТЕОРИЯ ОТ ЧАТА ЖПТ**

## 📡 Каналы в Go: что это?

**Канал (`chan`)** — это специальный тип данных, предназначенный для **синхронизации и передачи данных между горутинами**.

`ch := make(chan int)        // небуферизированный ch := make(chan int, 10)    // буферизированный`

---

## 🧭 Главные цели канала

- **Синхронизация**: передача значения блокирует отправителя или получателя до завершения операции.
    
- **Передача данных**: безопасный способ обмена значениями между потоками выполнения.
    

---

## 🧰 Типы каналов

### 1. 🧵 **Небуферизированный канал**

- Передача значения происходит **синхронно**:  
    `send` блокируется, пока кто-то не сделает `receive`, и наоборот.
    
- Под капотом — почти прямая передача: **одна горутина "будит" другую**.
    

### 2. 📦 **Буферизированный канал**

- Использует **встроенный буфер** (круговая очередь).
    
- Передача значений **возможна без немедленного получателя**, пока буфер не заполнен.
    

`ch := make(chan int, 4)`

---

## ⚙️ Внутреннее устройство буферизированного канала

Внутри буферизированного канала:

|Компонент|Назначение|
|---|---|
|`buf []T`|Круговая очередь — буфер|
|`sendx`|Индекс, куда писать следующее значение|
|`recvx`|Индекс, откуда читать следующее значение|
|`qcount`|Кол-во элементов в буфере|
|`closed`|Флаг: закрыт ли канал|
|`sendq`|Очередь горутин, ждущих отправки|
|`recvq`|Очередь горутин, ждущих получения|

### 🔁 Пример работы:

- Изначально `sendx = recvx = 0`
    
- `send` записывает в `buf[sendx]`, затем `sendx = (sendx + 1) % cap`
    
- `recv` читает из `buf[recvx]`, затем `recvx = (recvx + 1) % cap`
    
- **Если буфер полон**, отправитель **блокируется**
    
- **Если буфер пуст**, получатель **блокируется**
    

### 💤 Блокировка и пробуждение

- Заблокированная горутина **"спит" через `gopark()`**
    
- Когда канал становится доступным — **пробуждается через `goready()`**
    

---

## 🤝 Взаимодействие в небуферизированном канале

- Никакого буфера: передача только если другая сторона готова
    
- Реализуется через **парное связывание** горутин
    
- Под капотом `selectrecv`, `selectsend`, и очередь ожидания
    

---

## 📜 Аксиомы и поведение каналов

|№|Аксиома|Поведение|
|---|---|---|
|1|Чтение из **закрытого** канала|Возвращает **нулевое значение типа** + `ok = false`|
|2|Запись в **закрытый** канал|**panic** `send on closed channel`|
|3|Запись в **переполненный буфер** (или пустой небуфер)|Горутина **блокируется**, пока не появится место|
|4|Чтение из **пустого канала**|Горутина **блокируется**, пока не появится значение|

### Пример:

`ch := make(chan int, 1) ch <- 42 close(ch)  x, ok := <-ch // ok = true y, ok := <-ch // ok = false, y = 0`

---

## 🚦 Односторонние каналы

Можно ограничить доступ только к чтению или только к записи:

`var sendOnly chan<- int   // только запись var recvOnly <-chan int   // только чтение`

---

## 📊 Типичные применения каналов

- **Pipeline-паттерны**
    
- **Fan-out / Fan-in** схемы
    
- **Работа с `select` для multiplexing'а**
    
- **Семафоры, контроль параллелизма**
    
- **Блокировки через `chan struct{}`**
    

---

## 🔥 TL;DR — Основные тезисы

|Тема|Кратко|
|---|---|
|Канал|Синхронизирует горутины через передачу значений|
|Буферизированный канал|Имеет очередь (круговой буфер), блокирует при переполнении|
|Небуферизированный канал|Передача происходит напрямую между горутинами|
|`sendx`, `recvx`|Указатели внутри буфера|
|`gopark` / `goready`|Механизмы блокировки и пробуждения|
|Чтение из закрытого канала|Возвращает `zero value` + `ok=false`|
|Запись в закрытый канал|Паникует|
|Пустой канал|Блокирует операцию|
