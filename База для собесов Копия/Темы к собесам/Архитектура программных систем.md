## **Основные принципы чистой архитектуры**
Чистая архитектура основана на разделении системы на слои, где:

- **Бизнес-логика** (сущности и сценарии) находится в центре.
- **Внешние интерфейсы** (базы данных, API, UI) расположены на периферии.
- **Правило зависимостей**: зависимости направлены только **внутрь**.

## **Преимущества чистой архитектуры**

| Описание                                                        | Принцип                          |
| --------------------------------------------------------------- | -------------------------------- |
| Архитектура не зависит от существования конкретного фреймворка. | **Независимость от фреймворка**  |
| Бизнес-правила можно тестировать без UI и БД.                   | **Тестируемость**                |
| UI можно заменить без изменений в бизнес-логике.                | **Независимость от UI**          |
| Можно заменить SQL на NoSQL без влияния на бизнес-логику.       | **Независимость от базы данных** |
| Бизнес-правила не зависят от внешних сервисов.                  | **Независимость от сервисов**    |

---

## **Слои архитектуры**

Чистая архитектура представлена в виде **концентрических кругов**:

| Слой                      | Описание                                                    |
| ------------------------- | ----------------------------------------------------------- |
| **Сущности**              | Основные бизнес-объекты, наиболее стабильная часть системы. |
| **Сценарии**              | Определяют бизнес-логику и потоки данных.                   |
| **Интерфейс-Адаптеры**    | Связывают бизнес-логику с интерфейсами.                     |
| **Фреймворки и драйверы** | Окружение системы (БД, UI, API).                            |

### **Правило зависимостей**

Зависимости в коде **всегда направлены внутрь**. Внешние слои не должны знать о внутренней логике.

---

## **Критерии плохой архитектуры**

|Критерий|Описание|
|---|---|
|**Жёсткость**|Любые изменения требуют переделки всей системы.|
|**Неподвижность**|Код не разделён на переиспользуемые модули, сложно рефакторить.|
|**Вязкость**|Чрезмерная централизация или разбиение кода на мелкие пакеты.|
|**Излишняя повторяемость**|Дублирование кода вместо использования зависимостей.|

---

## **Сравнение Чистой архитектуры и Монолитного подхода**

|Критерий|Чистая архитектура|Монолит|
|---|---|---|
|**Гибкость**|Высокая, можно заменять модули без затрагивания системы|Низкая, любое изменение затрагивает всю систему|
|**Тестируемость**|Легко тестировать каждый слой отдельно|Тестирование сложное, так как компоненты связаны|
|**Расширяемость**|Можно добавлять новые компоненты, не изменяя старые|Новые функции требуют значительных изменений|
|**Разделение ответственности**|Чёткое разделение слоёв|Все компоненты связаны|

## **Пересечение границ слоёв**

Каждый слой общается с соседним только через **интерфейсы** и **DTO**. Данные не передаются напрямую между слоями, а преобразуются в удобный формат.

### **Как данные пересекают границы?**

- **Используются DTO (Data Transfer Objects)** – объекты передачи данных между слоями.
- **Не передаются сущности напрямую** – чтобы избежать утечки зависимостей.
- **Обратные зависимости решаются через интерфейсы**.
