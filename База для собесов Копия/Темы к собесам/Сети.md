## **Как работает запрос в браузере**


1. **Поиск в локальном кэше**
    
2. **Запрос к DNS-серверу** (проверка кэша, обращение к зонам .com и получение IP)
    
3. **Установление TCP-соединения**
    
4. **Отправка HTTP-запроса**
    
5. **Получение HTTP-ответа**

## **Протоколы и их описание**

- **Ethernet II** – основа передачи данных в сети
    
- **IP (Internet Protocol)** – маршрутизация
    
- **ICMP (Internet Control Management Protocol)** – сообщения об ошибках
    
- **UDP (User Datagram Protocol)** – быстрые, но ненадежные передачи
    
- **TCP (Transmission Control Protocol)** – гарантированная доставка
    
- **DHCP (Dynamic Host Configuration Protocol)** – назначение IP
    
- **DNS (Domain Name Service)** – преобразование доменов в IP

## **HTTP**

![[Untitled 3 4.png|Untitled 3 4.png]]

### **Основные методы HTTP**

- **GET** – получение ресурса
    
- **POST** – отправка данных на сервер
    
- **PUT** – обновление ресурса
    
- **DELETE** – удаление ресурса

- **PATCH** – частичное обновление
    
- **HEAD** – получение заголовков без тела ответа
    

![[Untitled 4 3.png|Untitled 4 3.png]]

### **Коды ответов HTTP**

- **1xx** – информационные
    
- **2xx** – успешные (200 OK, 201 Created)
    
- **3xx** – перенаправления (301, 302)
    
- **4xx** – ошибки клиента (400 Bad Request, 404 Not Found)
    
- **5xx** – ошибки сервера (500 Internal Server Error, 503 Service Unavailable)

## **TCP vs UDP**

**TCP/UDP** - это **транспортные протоколы**, используемые для данных передачи по сети.

| Функция                    | TCP (надёжный)             | UDP (быстрый)           |
| -------------------------- | -------------------------- | ----------------------- |
| **Гарантия доставки**      | ✅ Да (подтверждения `ACK`) | ❌ Нет                   |
| **Сохранение порядка**     | ✅ Да                       | ❌ Нет                   |
| **Повторная отправка**     | ✅ Да                       | ❌ Нет                   |
| **Задержка (ping)**        | ❌ Выше                     | ✅ Ниже                  |
| **Использование ресурсов** | ❌ Больше (из-за контроля)  | ✅ Меньше                |
| **Используется в**         | Веб, файлы, базы данных    | Игры, видео, голос, DNS |

### TCP

#### ==**База**==

- **Надёжный:** Гарантирует доставку данных, их порядок и повторную отправку в случае потерь.
    
- **Медленный:** Использует подтверждения, что добавляет задержки.
    
- **Используется:** Веб-сайты (HTTP/HTTPS), FTP, SSH, email.
    

#### UDP (User Datagram Protocol):

- **Быстрый, но ненадёжный:** Не гарантирует доставку и порядок данных.
    
- **Используется:** Видео/аудио стримы, онлайн-игры, DNS.
    

#### Гарантии TCP:

- **Подтверждения (ACK)**: Отправитель ждёт подтверждения от получателя.
    
- **Повторная передача**: Если подтверждение не пришло, данные отправляются снова.
    
- **Контроль ошибок и порядка**: Пакеты проверяются и упорядочиваются.
    
- **UDP** – без установления соединения, высокая скорость, подходит для видео/аудио потоков.
    



---

### UDP

#### ==**База**==

- **Быстрый, но ненадёжный:** Не гарантирует доставку, порядок или повторную отправку данных.
    
- **Меньше задержек:** Нет необходимости в установлении соединения и подтверждениях.
    
- **Используется:**
    
    - **Стриминг видео и аудио** (например, YouTube, Netflix, VoIP)
        
    - **Онлайн-игры** (например, Fortnite, CS:GO)
        
    - **DNS-запросы** (поиск IP-адресов)
        
    - **IPTV и сетевые трансляции**
        

#### Особенности UDP:

- **Нет подтверждений (ACK)** — получатель не отправляет подтверждения о получении.
    
- **Нет контроля порядка** — пакеты могут приходить в произвольном порядке.
    
- **Нет повторной передачи** — потерянные пакеты не отправляются снова.
    

#### Преимущества UDP:

- **Быстрота:** Отсутствие задержек на установление соединения и подтверждения делает его быстрее, чем TCP.
    
- **Меньше нагрузка:** Использует меньше ресурсов из-за простоты и отсутствия подтверждений.

## **gRPC**

![[Untitled 7 2.png|Untitled 7 2.png]]

- **Использует ProtoBuf вместо JSON** (меньший размер, быстрее обработка)
    
- **Работает поверх HTTP/2**
    
- **Поддержка потоков**
    
- **Бинарный формат передачи**
    
- **Кодогенерация для разных языков**
    

### **Типы gRPC вызовов**

```
// Обычный запрос-ответ
rpc Send(Message) returns (Result);

// Потоковый ответ
rpc Send(Message) returns (stream Result);

// Поток запросов, один ответ
rpc Send(stream Message) returns (Result);

// Поток запросов и ответов
rpc Send(stream Message) returns (stream Result);
```

## **Сравнение JSON и ProtoBuf**

![[Untitled 9 2.png|Untitled 9 2.png]]

- **JSON** – текстовый формат, легко читается, требует больше места.
    
- **ProtoBuf** – бинарный формат, компактный, требует кодогенерации.

## **REST API**

![[Untitled 10 2.png|Untitled 10 2.png]]

- **Клиент-серверная модель** – четкое разделение ролей.
    
- **Структурированные методы (CRUD)** – операции с данными через HTTP.
    
- **Кеширование** – уменьшение нагрузки на сервер.
    
- **Отсутствие состояния** – каждый запрос независим.

## **WebSocket**

![[Untitled 11 2.png|Untitled 11 2.png]]

- **Постоянное соединение** между клиентом и сервером.
    
- **Отправка данных в реальном времени** (чат, стриминг, биржи).
    
- **Быстрее HTTP** – нет необходимости устанавливать соединение на каждый запрос.
